# Data Central

The UPTIMIZE Data Central Platform is a Data Catalog solution which has the purpose to collect from multiple source systems metadata and make them transparent, findable, and reusable.   


## What data will you find in UPTIMIZE Data Central? 

Data Central scans and collects metadata from the source system, not the actual data of a source system. But what is metadata? Metadata is data about data, e.g., descriptions or table and column names, but also ownership, data classifications and the origin or relationship between data elements. In many cases, these details can be read automatically from the database schema of the source systems.

**The Search Engine**

The search engine uses a fuzzy search algorithm, which facilitates exploration. In other words, it will try to find all the Objects that correspond exactly to the keywords provided, as well as those that correspond to keywords close to those entered. This method allows you to be permissive concerning any typos in the search words or Object content, as in the description, for example.

**Search Engine Drop-Down Menu**

When you click on the search bar, a drop-down menu appears: 
- Search suggestions based on your recent activity or that of other users
- A list of advanced search operators: 
  - "INFIELD:": Allows you to limit the search to information specific to the Fields in your Datasets
  - "AND": Allows you to search for Objects matching all keywords


**Filters**

On the results page, you can use filters to refine your search and reduce the number of results returned. 
Different filters are available:
- Item Type: The type(s) of Items in the catalog (Datasets, Visualizations, etc.)
- Data Source: System from where Items have been harvested.
- Contacts: Contacts associated with the Items (Data Steward, Data Owner, etc.).
- and more
Filters are dynamically updated and sorted according to their relevance and impact on your search.

**Search results**

The following information is displayed for each result corresponding to your search: 
- Name
- Source 
- Start of description
- Main properties
- Associated Glossary Items

N.B. If the Item doesn't have its definition and is associated with a single Glossary Item. The latter's description is used.

**Item overviews**

On the results page, you can click on each result to open its overview panel and access the Item's main information. You can also navigate within this overview panel to other Items without losing your search context.
Recurring information on the overview panel includes, among others : 
- Technical name
- Source metadata
- Associated contacts
- Related Glossary Items

## Discover Catalog Items

All catalog Items (except Fields) have a details page, accessible from the search page or from an overview panel.
Detail pages include common elements (name, description, etc.) and others specific to the type of Item or your platform configuration (lineage, data model, etc.).

**Details Section**

![details](https://docs.uptimize.merckgroup.com/docs-assets/data-central/details_section.png)

**People:** This section is used to quickly identify the right contacts for an item in the catalog. 
Contacts may include natural persons, legal persons, or legal entities. For each contact, name, email address (if provided) and responsibility for the Item (Data Owner, Data Steward, etc.) are available.
The badge "Curator" defines who will oversee the documentation of the Item in Data Central.

**Description:** The description provides more business or technical context for the Item, such as its definition, usage, calculation formula, etc.
In general, this description is entered into Data Central Studio by the Item's Curator.
However, in the case of an Item harvested from a storage system (Datasets, Visualizations, etc.), a description directly provided by the source can be used.

**Glossary:** These terms are filled in from Data Central Studio by its users and bring a business outlook 

**Properties:** This section contains the information/properties managed by Data Central Studio. 

The featured properties can be of two kinds: 
- Business properties: they are filled in according to the metamodel configured in Data Central Studio
- Technical properties: these are linked to the source file

Some properties that are considered important by Data Stewards or Curators are given more prominence and are directly displayed as tags in the search or detailed pages.

From the Data Central Explorer search page, you can export documentation for all search results.

To export documentation for a batch of Objects:
- Run a search (from the search bar or by selecting a filter)
- Select the desired type of Object (Data Sets, Data Processes, etc.)
- Then click on the "Export" button to start preparing your file.

Your selection must comply with the following constraints: 
- The selected objects must all be of the same type (Dataset, Data Process, etc.).
- The number of Items selected must be less than 100,000.

**Note:** The export functionality depends on technical constraints set by Excel; as such, properties containing more than 32,000 characters may generate an error message, as they would then exceed the maximum number of characters per cell allowed by Microsoft Excel.
Once you've finished exporting, click on the "Download file" button to download the file (.xlsx format) to your computer.

The following information is exported: 

| **Field**|**Description**
|:----------|:-------|
|Key  |Contains the unique identifier for that object.
|Name  |Name of the Item
|Source Name  |Name as defined in the source (if the Item is imported from a connector)
|Description  |Description of the Item
|Source Description  |Description as defined in the source
|Contacts  | One column for each responsibility; curators are exported in the "curator" column
|Properties  | One column for each property
|Links with Custom Items or Glossary Items  | Two columns: one for the unique keys of linked Items and another for names of related Items (grey background)
|Inputs & Outputs (Data Process only)  | Two columns containing the keys of the Datasets or Custom Items linked to the Data Process
|Usability (for Datasets only)  | Usability information for Datasets
|Calculation of hte Data Profiling (fingerprinting)  | Only for Fields.
|Publishing of the Data Profiling  | Only for Fields.
|Type  | Type of the Item for Datasets (e.g. Table) and Fields (e.g. Integer)
|Primary Key  | Is the Field defined as a primary key (true/false)
|Foreign Key  | Is the Field defined as a foreign key (true/false)
|Can Be Null  | Can the Field be null (true/false)
|Multivalued | Is the Field multivalued (true/false)
|Native Type  | Type of the Field as defined in the source


Please note the following behaviors:

- Information from a connector cannot be modified with a file import (ex: source name, source properties, etc.). The corresponding columns are grayed in the export file to identify them easily.
- When an attribute (property, responsibility, etc.) is multivalued, the values will be separated by semi-colons.
- For URL-type properties, if a label have been defined, the format of the value will be the following: https://www.mycustomurl.com[label]
- For Glossary Items, the file will contain columns for the "parent" Item Types as defined in the Glossary metamodel. For instance, if the Glossary Item "Turnover" is linked to the parent Item "Finance" and a child Item "Overall retention rate" when exporting the Item "Turnover", only the link with "Finance" will be exported and not the one with "Overall retention rate".

## What are the Widgets in the Dashboard?

The Dashboard is the Studio's home page. It is made up of various graphical components or "widgets" enabling Data Stewards, or Curators, to monitor progress or facilitate the management of their scope of responsibility within the catalog.

### Last Search

This component lists your last 5 searches.

### Most Popular Items

This component presents a list of the most viewed Items in your perimeter over the last 30 days.

### Watchlists

Watchlists allow you to save your favorite filters and quickly find your lists of Items to process. This component lists your [Watchlists](https://support.zeenea.com/hc/en-001/articles/21262653509905). By default, it contains 5 Watchlists:
- Datasets for which you are the Curator and for which the completion rate is less than 50%.
- Glossary Items for which you are the Curator and for which the completion rate is less than 50%.
- [Orphan](https://support.zeenea.com/hc/en-001/articles/19838687460241) Datasets for which you are the Curator
- Fields for which you are the Curator and for which suggestions are pending. These are Fields that may contain Personally Identifiable Information. The method for detecting and suggesting these Fields is described [here](https://support.zeenea.com/hc/en-001/articles/21263451973137).
- Items for which you are Curator and for which user suggestions are pending

### Perimeter

This component shows, by Item type, the number of Objects in your perimeter, as well as their completion rate, according to the rules defined by your Administrators. For more details, see [How the documentation completion level is calculated](https://support.zeenea.com/hc/en-001/articles/21262017382417).

## How the Documentation Completion Level is Calculated

Zeenea offers, using a graphic component, an indicator of the documentation's level of completion at a given moment in time.

This indicator is calculated for each Item and is displayed in the search results and on the Item detailed page.

The completion level is also calculated for the perimeter of a Curator (i.e. all the Items assigned to it) by Item Type. It can be viewed on the Dashboard.  

The calculation of the completion level is based on 4 criteria:
- Is the description filled in?
- Has at least one contact been added?
- Has at least one Glossary Item been linked (this condition is ignored if the metamodel does not support this type of link)?
- Are all the "Standard" and "Important" properties filled in?

Descriptions from the source are also taken into account in the calculation of the completion level.

In case of an update of the metamodel (i.e.: a property is removed from the template), the completion level is automatically updated. Please note that it is an asynchronous task that can take several minutes.

An administrator can define which properties are "Standard" or "Important" in the Catalog Design section. You can find this information in the tooltip on the details page of each Item


## Show the Data Lineage with Data Processes

By default, Data Processes can be linked to Datasets imported into the catalog to form a lineage of your data.

In the Catalog Design section, you have the possibility to create additional links in your metamodel with Custom Item Types, in order to complete your lineages with non-synchronized Items or to represent lineages at a logical level.

To do this, click on the "Edit Item Type" button (pen icon) in front of the Data Process type to open the link editing mode.
 
In this mode, select the Custom Item Types with which you wish to allow Data Stewards to create links as inputs to your Processes. Then select those for which you wish to create output links.

**Note:** It is impossible to delete the link with the Dataset type. To remove a link with a custom type, first, make sure that there are no more links in your catalog between Treatments and Objects of the selected type.

## Foreign Keys and Business Keys

### Primary Key

The Primary Key is the dataset technical identification key. It is defined by the system and cannot be manually edited.

For example, the "Customer Number" could be a Primary Key in the "CUSTOMERS" table.

### Foreign Key

The Foreign Key is a reference to a Primary key of another. It is defined by the system and cannot be manually edited. 

For example, the "Customer id" could be a Foreign Key in the "ORDER" table, referencing the Primary Key "Customer Number" in the "CUSTOMERS" table.


### Business Key

A Business Key (aka Natural Key) is any combination of Fields that would allow people to instinctively identify a piece of data in the dataset. 

In our "CUSTOMER" table example, the combination of last_name/first_name/birthdate could be a Business Key. 

The Business Key is defined by Data Stewards in the catalog. 


### Visualizing Keys and Their Underlying Fields

#### From a Dataset:

Keys can be retrieved from the "Fields" tab of a dataset. 

In that list, a specific icon will identify the type of key each field is a part of (BK, PK, FK).


#### From the Lineage:

Fields composing Primary, Foreign, and Business Keys can be seen from the lineage tab. You will need to expand the view and list all fields in the lineage to do so; an icon is used to identify the type of key each field is a part of.  

#### In the Search Results:

Keys have been added to Search Results as well as the Overview panel, in order to give more context at a glance. 

This information is displayed as an icon in the Search Suggestions, the Search Results, and Items overview panel. 

### Defining the Business Key

To define a Dataset Business Key, head to the "Fields" tab. You'll then be able to select all fields composing a Natural Key by adding them to the dedicated list

You can also remove a field from a Business Key by clicking on the "Delete" icon. 

**Note:** the Primary Keys and Foreign keys are defined in the source system and cannot be manually edited.

**Note 2:** the Business Keys cannot be updated via an Excel import or a bulk edit.

### Requirements

To import Datasets or Visualizations  into Zeenea, you must first: 
1.	Install a scanner 
2.	Install the plugin of the connector that is appropriate to your storage system
3.	Configure a connection to your storage system
4.	Launch an inventory of this connection to allow the connector to automatically discover the Items available for import.

### Import Datasets or Visualizations

### Step 1: Connection

First, select the connection corresponding to the items to be imported.

Some specifics about this step: 
- Only connections that work with an inventory are proposed, namely those of the "dataset" or "visualization" type. The other "synchronized" connections are automatically imported in an all-or-nothing mode. 
- Connections with automatic import activated are grayed out. If necessary, it is possible to force an automatic import from the administration interface.


### Step 2: Items

Search for the items either:
- by browsing the folders (by clicking on their name) 
- by using the search bar

Select the items to import:  
- individually: Click on the cross next to an item 
- in bulk: Click on "Add all" to add the items in a folder to your import 
Tick the "Show already imported" box to view the items already imported in the catalog.


### Step 3: Curators

You can now add one or more Curators to datasets when importing them. These will be the Stewards in charge of documenting the imported datasets. 

This step is optional. 

**Note:** If the user requesting the import does not have permission to edit unassigned items, he will be automatically set as a curator for these newly imported items.

### Step 4: Contact

It is possible to associate a contact and a responsibility to the datasets resulting from your selection. 

This step is optional.

### Step 5: Category

For datasets only, it is possible to specify a category among those already created.

This step is optional.

### Step 6: Confirm

Before final confirmation, a summary of the operations to be carried out is proposed in the last stage.

IMPORTANT: After confirmation, the window is closed and the import is performed asynchronously. The operation may take several minutes depending on the number of items selected.


### The Specifics of Certain Imports

For datasets:
- For these imports, the fields are also imported automatically. Once associated with the source dataset, the fields also become searchable items in Zeenea.
- For some connections, the automatic import of the Data processes associated with this dataset is possible (ex: Atlas)

For visualizations: 
- The import of visualizations, if the connection allows it, can also include the associated datasets and their fields.

The imported items appear progressively in the Catalog section. Once the import is completed, a notification is sent with the number of imported items and the number of failures, if any. This notification makes it easy to find the list of imported items.

# Automation Rules by Property
This section explains how metadata automation works across Data Central and connected UPTIMIZE systems. It shows which properties are overwritten, conditionally replaced, merged, or controlled by the system for sources such as Foundry Data Treasury, UPTIMIZE Harbor, UPTIMIZE Vector Database, SAP BW, and SAP SuccessFactors.
The goal is to help Data Stewards, Data Owners, and Producers understand which metadata they can maintain manually and which values are updated automatically.

| **Rule** | **Description** | **Applicability** | **What Users Can Expect** |
|---------|--------------------------------|-------------------|---------------------------|
|Foundry Data Products sync automatically | Key properties are overwritten by Foundry. | Data Products with *Data Source = Foundry Data Treasury* | Manual edits to these fields will not stay. |
|Data Domain & Organizational Unit are merged | Your values stay; parent items are added automatically. | All item types supporting Domain/Organizational Unit| Additional parent links may appear automatically. |
|SAP BW dataset names may update | Display Name updates to SAP BW business name if never manually changed. | SAP BW datasets | After you rename it once, automation stops changing it. |
|Dataset classification inherits from fields | Dataset takes the highest field classification. | Datasets with empty classification | One Secret field makes the dataset Secret. |
|Harbor & Vector DB overwrite metadata | Harbor overwrites System of Record; Vector DB overwrites Description, PII, Identification Classification, GxP. | Harbor & Vector DB datasets | Manual edits to these fields will not persist. |
|Foundry datasets inherit Contacts from DP | Dataset Contacts sync from the linked Data Product. | Foundry DTI datasets | Manual Contact changes will be overwritten. |
|Contacts may become Curators | Contacts are added to Curators for Studio users. | All item types with Contact | Contacts may appear under Curators. |
|Field names may update from source | Display names update to business/logical name unless manually changed. | SAP BW & SuccessFactors fields | Manual renaming prevents future overwrites. |

# Capabilities & Concepts

UPTIMIZE Data Central offers a wide range of capabilities. A detailed list can be found on this page: 

 

**Data Discovery:** Finding relevant data has never been easier; users can filter and categorize search results to refine their queries.  
Strongly inspired by marketplaces and e-commerce websites, UPTIMIZE Data Central offers smart searching capabilities for everyone at Merck, helping users streamline decision-making processes and reduce the time spent on data retrieval.
 

**Data Lineage:** Trace your data transformations through automatic lineage. 
UPTIMIZE Data Central's Data Lineage capabilities enables you to get a complete view of your data's lifecycle - from its collection to its use, storage, and preservation over time. Automatically map the relationships between systems, applications, and reports to provide a context-rich data ecosystem for your organization. Through an interactive and user-friendly lineage graph, better understand your data, easily identify transformations. 

 

**Business Glossary:** Enable data literacy with a business glossary. 
UPTIMIZE Data Central's Business Glossary features enable the creation and sharing of a consistent data language across all data consumers within Merck.  

 

**Metadata Management:** Leverage your data assets with active metadata. 
What is Active Metadata? Active metadata that is continuously updated and actionable, allowing real-time insights into data usage and performance. UPTIMIZE Data Central provides an active metadata management solution to power your data & analytics use cases. Our platform collects metadata from various sources, organizes it in a centralized repository, and makes it accessible across the organization. With UPTIMIZE Data Central, you will spend less time searching for information and more time getting value from data, enabling you to drive better business decisions.
 

**Data Stewardship:** Increase data knowledge by automatically capturing and updating metadata in real-time. 
UPTIMIZE Data Central provides a metadata management solution that enables Data Stewards to overcome the challenges of handling increasingly large volumes of data, ensuring better data accuracy, governance, and compliance. Our platform helps organizations maximize the value of their data by reducing the time spent on complex and time-consuming documentation tasks. It also breaks down data silos by creating centralized access to dispersed data sources, increasing enterprise-wide data knowledge.
 

**Data Governance:** Effectively manage risks, ensure privacy, and comply with data regulations. 
UPTIMIZE Data Central provides a solution that enables you to find, understand, and create a common language around data by standardizing data definitions and terminologies used across the organization. It contributes to an active and agile data governance system that delivers trusted, compliant, and secure information across your organization. Through smart recommendations, our platform detects personal information and suggests which assets to tag--ensuring that data policies and regulations are well communicated to all data consumers in their daily activities.

**Data Quality:** Data Central enables you to synchronize quality information from third-party solutions relating to your Datasets with the catalog. Thanks to this synchronization, you can check the quality level of your Datasets according to the criteria and controls defined in your Data Quality Management tools directly from your Studio and Explorer applications.

# Data Central Product Development Approach
The Data Central Product Team - the Data Lingos - are an integrated team of the UPTIMIZE Agile Release Train and is working as per the Scaled Agile Framework (SAFe).

## How do we work within UPTIMIZE
We operate on two different levels. On program level the strategical planning is done by the stakeholders in close collaboration with the cross functional *Agile Release Team (ART)* in a 3-month cycle. On team level the teams work in shorter cycles of two weeks and continuously working on the development of features for the UPTIMIZE Ecosystem.

You can find more details about the UPTIMIZE Release Train and its working model [here](https://docs.uptimize.merckgroup.com/pom/)

## Goal

- Making our Data Central service resilient to incidents  
- Making our work efficient  
- Making our service manageable for other teams(-members)  
- Fostering culture of continuous feedback & communication between all core team members  

Indicators:  

- Anyone in the team can take over another team mates tasks with minimal effort and finds all the relevant documentation on implementation, reasoning, and decisions 
- A new team or team member can take over with minimal effort 


## Program Level Events
The **PI planning session**, which takes place every 3 months, marks the start of a **Program Increment (PI)**, and determines the desired features for the next weeks. The output is a delivery plan that will be implemented by the teams during the PI, which is typically 5 Iterations long. The Program Level Events/Meetings are:
- PI Planning sessions - every 3 months, recording of the PI Planning-04
- ART (Product Owner & Coach) sync - weekly.
- System Demo - after 2 iterations,
- Inspect & Adapt Workshops - every 3 months.
- I&A Quantitative and Qualitative Measurement Session - every 3 months.
- SAFe for Teams Training - every 3 months (for the new joiners of the project).



## Team Level Events
As UPTIMIZE is implemented in an agile framework, the teams are self-organized and work iteratively on the features in so called Sprints/Iterations. A Sprint starts with a Sprint Planning Meeting to decide which items can be developed in the next iteration. One Iteration is 2 weeks long and contains following events:
- Sprint planning (beginning of each sprint)
- Daily Huddles/Stand-up
- Sprint Review (end of a sprint)
- Backlog Refinement (in the middle of the sprint)
- Retrospective (end of a sprint)
- Stakeholder meeting (every month)
- System Demo - after 2 iterations, recording of the System Demo PI4

# Roles & Responsibilities

UPTIMIZE Data Central is used by different user groups, which can be grouped into **Data Consumers** and **Data Producers**. <br>

**1: Initial Release of Harbor Snowflake Connector (QA - Beta)**

**What's New**

**Applicable Item Types**  
- Datasets

**Feature Overview**

**Current Behaviour (Production)**  
- This feature is not yet available in Production.

**Current Behaviour (QA Environment)**  
- Users can search and discover datasets in the catalog that are linked to the **UPTIMIZE Harbor Snowflake** connection.  
- Only datasets scanned via this connection are displayed.  
- In this initial release, datasets are imported exclusively from the SAP TEMPO ERP source system.  
- Each source system may have multiple system identifiers, selectable via the Instance filter.

**Benefits**  
- Unified Connection: Consolidates multiple ERP source systems under a single connection (UPTIMIZE Harbor Snowflake).  
- Enhanced Discoverability: Enables easy identification of datasets associated with a specific connection.  
- Improved User Experience: Reduces the effort required to locate related datasets.

**Future Enhancements**  
- Integration of additional source systems, automatically added during connection refreshes.  
- Enablement of downstream system linking for consuming datasets.

**Action Required**  
- **Testing:** Complete QA testing and provide feedback within 2 days.  
- **Feedback:** Report issues, suggestions, or concerns.  
- **Production Deployment:** Planned after review and validation of feedback.

Dataset and field name updates for SAP BW are now automated using Glue Jobs, replacing previous manual script-based updates.

**Applicable Item Types**  
- Datasets (SAP BW)  
- Fields (SAP BW)

**Feature Overview**

**Current Behaviour (Before Automation)**  
- Name updates were performed manually using scripts on a local system.  
- Newly synchronized SAP BW items required additional manual intervention.

**Current Behaviour (After Automation)**  
- Name updates are automated using Glue Jobs.  
- Updates are applied only when the item name does not match the Logical Name source property.  
- Automation runs on a monthly schedule, preferably during weekends.  
- Newly synchronized SAP BW datasets and fields are updated automatically when applicable.

**Benefits**  
- Eliminates manual intervention and operational overhead.  
- Ensures consistent naming across SAP BW datasets and fields.  
- Improves overall data quality and governance.

**Action Required**  
- Validate automated name updates in QA and Production.  
- Confirm correct behavior for newly synchronized SAP BW items.

# Frequently Asked Questions
## General FAQs

<details>
<summary style="font-size: 1em; font-weight: 500;">What is a data catalog and what is UPTIMIZE Data Central?</summary>

&nbsp;  
UPTIMIZE Data Central is Merck's data catalog. A data catalog is an organized inventory of data assets helping to collect and arrange metadata as well as enriching the information and making it findable, accessible, interoperable, and reusable. This facilitates quick and easy data discovery and management. UPTIMIZE Data Central contains information about Merck's data (for instance, data descriptions, relevant points of contact, data lineage, etc.), but not the data itself. The information in UPTIMIZE Data Central allows data consumers to confirm a particular dataset will help them answer their business questions and then provide instructions on how to request access.

</details>

<br>

<details>
<summary style="font-size: 1em; font-weight: 500;">What is the benefit of a Merck-wide data catalog?</summary>

&nbsp;  
A data catalog will enable us to search easily and effectively, access and use required data. It will also allow for standardized data definitions with pre-curated data assets and an overview of data inventory and data legacy. Data producers and data consumers will benefit by being able to easily access metadata for data sources across a variety of systems at Merck.

</details>

<br>

<details>
<summary style="font-size: 1em; font-weight: 500;">How is the data catalog related to other data initiatives at Merck, such as Data Treasury, data Governance, Foundry, and data semantics?</summary>

&nbsp;  
The data catalog team is in close contact with other initiatives that relate to data quality and data transparency at Merck. Together, all of these initiatives will make for a seamless experience for data consumers and producers. Data Treasury will serve as one of many inputs to the data catalog, UPTIMIZE Data Central. UPTIMIZE Data Central will not replace Treasury, but rather will build on top of it, as other source systems (such as SAP BW) will also be brought into the catalog.

</details>

<br>

<details>
<summary style="font-size: 1em; font-weight: 500;">Why was Zeenea selected for Merck's data catalog?</summary>

&nbsp;  
Zeenea stood out to the team due to its user-friendly interface and intuitive functionality. We know that a key factor to the data catalog's success is user adoption and ensuring data sources are connected, so it was important to prioritize a modern, user-friendly solution over a more complex tool. We are also happy with Zeenea's roadmap and Merck's ability to partner with them and help shape that roadmap further.

</details>

<br>

<details>
<summary style="font-size: 1em; font-weight: 500;">Who should use the data catalog?</summary>

&nbsp;  
Everyone at Merck who works with data can benefit from a data catalog by having greater transparency into what data is available at Merck. Data producers can use the catalog to showcase information about data in their space while data consumers can browse what is available with easy-to-use search functionalities.

</details>

<br>

<details>
<summary style="font-size: 1em; font-weight: 500;">Can the data catalog help in identifying data quality issues and data anomalies?</summary>

&nbsp;  
A data catalog can help make bad data quality transparent, but ultimately, the data quality needs to be addressed at the data source. Out of the box, Zeenea does not have a data quality module, however, there is the ability to integrate third-party solutions. This integration is not in our immediate deployment plan for the catalog.

</details>
